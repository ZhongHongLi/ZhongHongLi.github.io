# git的三个作用域

pwd 现有目录的当前路径

$ git config --local local 只对某个仓库有效

$ git config --global global 对当前用户的所有仓库有效

$ git config --system system 对系统所用登录有效

显示配置config的配置，加 --list $ git config --list --local

$ git config --list --global

$ git config --list --system

$ git config --global --list 查看用户名跟邮箱

建git仓库 两种场景 1.把已有的项目代码纳入git管理 $ cd项目代码所在的文件夹 $ git init

2.新建的项目直接用git管理 $ cd 某个文件夹

$ git init your_project # 会在当前路径下创建项目名称同名的文件夹

$ cd your_project

# 集中式的工作流

就是在master 上直接开发,直接git clone --地址 在各自的模块开发先提交后推送，后面小伙伴更新一下在提交推送要不然提交不成功 适合3-5人开发

# git Flow 工作流

  • GitFlow 工作流定义了一个围绕项目发布的严格分支模型，虽然比功能分支工作流复杂积分，但提供了 用于一个健壮的用户管理大型项目的框架

  • GitFlow工作流没有超过功能分支工作流的概念和命令，而是为不同分支分配一个很明确的角色，并定义分支之间如何什么时 候进行交互，除了使用功能分支，在做准备，维护和记录发布也使用各自的分支，当然你可以用上功能分支工作流好处：pull Requests,隔离实验性开发和更高效的写作

    # 工作方式

  • Gitflow 工作流任然用中央仓库作为所有开发者的交互中心，和其他的工作流一样，开发者在本地并push分支到中央仓库中

    # 分支

    相对使用权有一个master分支，Gitflow工作流使用2个分支记录项目的历史master分支存储了正式发布的历史，而devlop分支作为功能的集成分支

    剩下要说明问题围绕着2个分支的区别展开

    # 功能分支

    每个新功能位于一个自己的分支，这样可以push到中央仓库以备份和写作，但功能分支不是从master分支上拉出新分支，而是使用devlop分支作为父分支，当新功能完成时，合并回devlop分支。新功能提交应该从不直接与master分支交互

    注意从各种含义和目的上来看，功能分支加上devlop分支就是工作流的用法，但GitFlow工作流没有再这里止步

    # 发布分支

    一旦devlop分支上有了做一次发布(或者说快到了既定的发布日)的足够功能，就从devlop分支上fork一个发布分支，新建的分支用户开始发布循坏，索易这个时间开始之后新的功能能不能再加到这个分支--这个分支只应该做 bug修复,翁当生成和其他面向发布任务，一旦对外发布的工作都完成了，发布分支就合并到master分支并合并一个版本号打好Tag,另外，这些从新建分布分支以来的做的修改要合并devlop分支

    使用一个用户发布准备的专门分支，使用一个团队可以在完善当前的发布版本的同事，另一个团队可以继续开发下个版本的功能，这也打造一定良好的开发阶段(比如，可以很轻松地说)

    • 常用的分支约定：
    • 用户新建发布分支的分支:devlop
    • 用户合并的分支：master
    • 分支命名：release- 或release/

    # 维护分支

    维护分支或硕士热修复(hotfix)分支用户生成快速给产品发布版本(production releases)打补丁，这是唯一可以从master分支fork出来的分支，修复完成，修改应该马上合并回master分支和devlop分支(当前的发布分支)，master分支应该用新的版本号搭好Tag

    为Bug修复使用专门分支，让团队可以处理掉问题而不用打断其他工作是等待下一个发布循坏，你可以把维护分支想成是一个直接在master分支上处理的临时发布

    常用的也就是集中工作流 还有一个就是git FLow工作流

# Gitlab·Gitflow工作流

  • 基本流程

1.项目组长创建仓库git flow init 2.成员进行clone git clone 3.项目组长分配里程碑与议题（任务） 4.项目成员根据自己被分配到的任务建立对象的分支git flow feature satrt xxx

  • 先在项目中建立分支，再拉取开发
  • 先在贝蒂建立分支，在提交 5.完成任务后pull request git push origin 6.pull request删除分支，关闭对应的议题任务git flow feature finish xxx
